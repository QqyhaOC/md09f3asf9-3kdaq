問題1: 不正解

ある企業はオンプレミス環境のインフラストラクチャをAWSに接続する計画をしています。その際は、低レイテンシーで可用性の高い安全な接続方法を必要としています。また、プライマリ接続障害時に冗長性を担保することが要件となっていますが、コストを最小限に抑えることが必要です。したがって、セカンダリの接続設定ではトラフィック転送速度のパフォーマンス低下は許容できます。


この要件を満たすために、ソリューションアーキテクトはどうするべきでしょうか。

説明

このシナリオでは、オンプレミス環境とAWSネットワークとの接続方法が問われています。その際に「プライマリの接続設定では低レイテンシーで可用性の高い安全な接続とすること」「セカンダリの接続設定では、コスト最適にプライマリ接続障害時に冗長性を担保しつつも、トラフィック転送速度のパフォーマンス低下は許容されること」が要件となっています。オンプレミス環境とAWSネットワークを接続する際は、AWS Direct Connectによる専用線接続を構成するか、AWS Site-to-Site VPNによるVPN接続を実施します。どちらを選択するかを要件に基づいて検討できる知識が問われます。


オプション１が正解となります。「プライマリの接続設定では低レイテンシーで可用性の高い安全な接続とする」という要件に従うためには、AWS Site-to-Site VPNによるVPN接続ではなく、AWS Direct Connectによる専用線接続を構成する必要があります。AWS Direct Connectはネットワーク回線を利用するAWS Site-to-Site VPNに比較すると、専用線接続による低レイテンシーで安全な接続が可能です。

AWS Direct Connect は50 Mbps から 100 Gbps までの速度で高性能な専用線接続を構成するサービスです。専用線をDirect Connectロケーションという中継地点に配置することで、直接にAWS側と物理的に接続することで高速なネットワーク接続を可能にします。



さらに今回のシナリオでは、セカンダリのネットワーク接続を構成して、オンプレミス環境とAWSネットワークとの接続構成を冗長化する必要があります。このセカンダリの接続設定では、要件に応じて、AWS Direct Connect を二重化する方法と、AWS Direct Connect に対してVPN接続を追加する方法を選択します。今回は「コスト最適にプライマリ接続障害時に冗長性を担保しつつも、トラフィック転送速度のパフォーマンス低下は許容されること」が要件となっています。したがって、性能よりもコストを優先するため、VPNを選択します。

AWS Direct Connect に対してセカンダリの接続設定としてIPsec (IP セキュリティ) を使用したVPC接続を追加することができます。VPN接続はDirect Connectに比較してトラフィック転送速度が遅いため、パフォーマンスが低下しますが、コストは割安となっています。


オプション２は不正解です。別リージョンのVPCに対してVPN接続をバックアップとして構成しても、メインで利用しているリージョンにおいて、オンプレミス環境とAWSネットワークとの接続構成を冗長化することにはなりません。別リージョンのAWSネットワークとの接続構成をする目的は、別リージョンでリソース展開をするケースや、別リージョンにおいて災害復旧構成を要する場合です。今回はそういった要件は求められていないため、同じリージョンにセカンダリ接続を構成する必要があります。

オプション３と４は不正解です。 Direct Connectの接続ラインを二重化する方法も可能ですが、これは最もコストがかかる方式です。コスト最適にプライマリ接続障害時の冗長性を担保しつつも、トラフィック転送速度のパフォーマンス低下は許容されるという要件に合致していません。

ある会社は、AWSを利用してモバイルアプリケーションを構築しています。このモバイルアプリケーションのセキュリティ要件では、ユーザー一人一人がMFA認証を利用してアプリケーションにアクセスすることが求められています。

この要件を満たすための、ソリューションはどれでしょうか。

説明

このシナリオでは、AWS上で開発しているモバイルアプリケーションにおいて認証機能を実装する方法が問われています。その際は、アプリケーションのユーザーの一人一人がMFA認証を利用してアプリケーションにアクセスすることが要件となります。


オプション３が正解となります。Amazon Cognitoを利用して構築しているアプリケーションの認証機能を実装することができます。Amazon Cognitoを利用することで、Google、Facebook、Amazon などのソーシャル ID プロバイダーや、SAML による Microsoft Active Directory などのエンタープライズ ID プロバイダーを通してサインインすることができます。そして、Amazon Cognitoの認証機能では、認証時のパスワード設定に加えて、多要素認証（MFA）を追加することが可能です。



オプション１は不正解です。IAMポリシーはAWS内のユーザー管理用のサービスであり、構築したアプリケーションの顧客管理機能として利用することはできません。

オプション２は不正解です。API GatewayやLambda関数はサーバレスなアプリケーションを実装して、APIからアクセスするための機能です。これを利用して、MFA認証を実装することはできません。

オプション４は不正解です。CloudHSMはクラウドベースのハードウェアセキュリティモジュール (HSM) です。これにより、AWS クラウド側で安全性の高いハードウェアベースの暗号化保存機能を利用して、暗号化キーを簡単に生成して使用できるようになります。CloudHSMによってMFA認証を実装することはできません。

説明

このシナリオでは、リレーショナルデータベースを利用したデータ処理ソリューションの最適な設定方法が問われています。その要件としては、「I/O性能を確保するための改善をすること」および「なるべく運用負荷をかけないデータベースを利用すること」が求められています。


オプション４が正解となります。このシナリオでは、リレーショナルDBを利用することが求められており、Amazon RDSまたはAmazon EC2インスタンスベースのリレーショナルデータベースが必要となります。その中では運用の手間をかけさせないという要件から、マネージド型で提供されているAmazon RDSの方が最適です。EC2インスタンスにインストールして構成するリレーショナルデータベースは、マネージド型で提供されないため、設定や運用に労力を要します。したがって、なるべく運用負荷をかけないという要件を達成できません。



「I/O性能を確保するための改善をすること」に対しては、Amazon RDSのインスタンスタイプとストレージタイプの選択において、IO性能が高いタイプを選択します。これによって、ランダムI/O遅延を防ぐことができます。

db.m5d は低レイテンシー、非常に高いランダム I/O パフォーマンス、高いシーケンシャル読み取りスループットに向けて最適化された汎用インスタンスクラスです。このDBインスタンスにこのクラスを利用することで、ランダムI/O遅延が発生しないパフォーマンスを発揮できます。

プロビジョンド IOPS ストレージは、低 I/O レイテンシーおよび一貫した I/O スループットが必要となる I/O 負荷の高いワークロード (特にデータベースワークロード) のニーズを満たすように設計されています。




オプション１は不正解です。DynamoDBはI/O性能が高いNoSQL型のデータベースの代表的なAWSサービスですが、リレーショナルデータベースをエミュレートしてリレーショナルデータとして利用できるように構成する機能は有していません。

オプション２は不正解です。Amazon ElastiCacheはAmazon RDSに連携して、クエリ処理を高速化することは可能です。今回はランダムI/O遅延が発生しているため、そもそものデータベース性能が不足していることが原因です。したがって、RDSのデータベース性能をI/O処理に適したものにする方が優先されます。

オプション３は不正解です。EC2インスタンスにおいてIOPS性能が高いインスタンスを設定して、データベースに利用することも可能ですが、マネージドサービスではないため運用負荷が高くなります。

説明

このシナリオでは、AWS上のアプリケーションのデータ処理状況を監視して、データ処理が停止した場合には担当者のモバイルに即座に通知されることが求められています。AWSではアプリケーション処理の異常正常を監視するためには、Amazon CloudWatchの機能を利用することになります。そして、Amazon CloudWatchにおいて通知機能を設定する際はAmazon SNSを利用することが一般的です。したがって、この質問ではこの２つのサービスのうまい組み合わせを選択することが問われています


オプション２が正解となります。 Amazon SNS を使用して、SMS 対応デバイスに SMS メッセージを送信できます。SMS メッセージは電話番号に対して直接メッセージを発行することが可能です。その際にAmazon SNS トピックに電話番号をサブスクライブする必要はありません。

オプション３が正解となります。Amazon CloudWatchアラームを利用してアプリケーションやEC2インスタンスの状態をトリガーとして、SNSメッセージを送信させることができます。したがって、停止イベントをトリガーとしてEメールを即時に送ることで通知することができます。

オプション４は不正解です。アラーム設定によりEメールを通知する場合はAmazon SNSを利用します。Amazon SESはEメール機能をAWS上で提供するサービスであって、これはアプリケーション機能としてメッセージ通知機能を実装する際に利用します。Amazon SESによって、アプリケーションの利用ユーザーに対して一斉メールによるお知らせを実施するといった機能を実装することが可能です。

オプション５は不正解です。メール通知はSNSを利用することが望ましいです。また、処理停止イベントをトリガーとするにはAmazon CloudWatchアラームが必要となります。Lambda関数だけで全機能を効率的に開発することはできません。

オプション３が正解となります。S3バケットの標準ストレージにデータを保存して、Amazon Athenaを利用してデータ解析することで、コスト最適なデータ分析ソリューションを実現することができます。Amazon AthenaはAmazon S3 バケットに保存されているデータに対して直接にクエリ処理を実施するサーバレスなデータ分析サービスです。その際には実行クエリに対してのみ料金が発生し、各クエリでスキャンされるデータ量に基づいて課金されます。データの圧縮、分割、列形式への変換を行うと、大幅なコスト削減とパフォーマンス向上を実現でき、データ処理コストを抑えることができます。

さらに30日が経過するとこれらのデータは必要がなくなるため、S3バケットに対してライフサイクルポリシーを設定して、30日後にデータを削除するように設定します。それによって、データ保存コストを最適化することができます。



オプション１は不正解です。Amazon RDSは一般的なリレーショナルデータベースとしてデータを処理するために利用されます。データを保存した上でのライフサイクル管理には向いていません。

オプション２は不正解です。Amazon DynamoDBはNoSQL型のデータベースであるため、SQLクエリ処理は実行できません。

オプション４は不正解です。Amazon S3 Glacier Instant Retrievalはデータの中長期保存用のアーカイブストレージタイプですが、瞬時にデータを取り出すことができる特殊なストレージです。滅多に利用することがないデータですが、データを利用する際は即時に利用する必要があるデータに適用します。例えば、過去の医療記録などです。しかしながら、企保的にはアーカイブデータであるため、保存したデータを分析するのには適していません。また最低保持期間が９０日と設定されているため、３０日間でデータを削除する場合では、コストが余計にかかるようになっています。

オプション３は不正解です。カスタマーゲートウェイはオンプレミス環境のネットワークとVPCを接続する際に利用するゲートウェイです。VPC内外のリソース間接続には利用されません。

オプション４は不正解です。プライベートサブネットからインターネットゲートウェイを介して、S3バケットにアクセスすることはできません。インターネットゲートウェイへのルートがある場合は、それはパブリックサブネットになります。また、インターネットゲートウェイを介してしまうとインターネット通信を利用することになります。

あるソリューションアーキテクトは新規にAWS上でアプリケーションを開発するために、ネットワークを設計しています。このネットワークは１つのVPCに対して、パブリックサブネットとプライベートサブネットを２つのアベイラビリティゾーンに１つずつ、合計で4つのサブネットを構成します。VPCとサブネットはIPv4 CIDRブロックを使用しています。


インターネットゲートウェイを使用して、パブリックサブネットにインターネットアクセスを接続しています。プライベートサブネットには、Amazon EC2インスタンスがソフトウェアアップデートをダウンロードできるようにインターネットにアクセスできる必要があります。その際は、データセンター障害に強い構成にする必要があります。


この要件を満たすために、ソリューションアーキテクトはどうすればよいでしょうか。

説明

オプション１が正解となります。このシナリオでは、プライベートサブネットにあるAmazon EC2インスタンスがソフトウェアアップデートをダウンロードできるようにするためのVPCのネットワーク設定が必要です。また、その際はデータセンター障害に強い構成にする必要があります。

プライベートサブネット内にあるインスタンスがインターネットへの通信をするには、パブリックサブネットにNATゲートウェイを配置する必要があります。その上で、プライベートサブネットのルートテーブルに対してNATゲートウェイを通じたインターネットへの返信のみを許可するルートを設定します。これによって、プライベートサブネット内のDBインスタンスへのパッチ適用や更新が可能となります。

各AZのパブリックサブネットに１つずつNATゲートウェイを作成することで、１つのアヴェイラビリティゾーンが停止しても、NATゲートウェイからのアクセスを継続するため、冗長性を担保することができます。



パブリックサブネットに設置されたEC2インスタンスはアウトバウンドトラフィックを直接にインターネット側に送信できますが、プライベートサブネットに設置されたEC2インスタンスはそれができません。代わりに、プライベートサブネットに設置されたインスタンスは、パブリックサブネットに設置されたネットワークアドレス変換 (NAT) ゲートウェイまたはNATインスタンスを使用して、インターネット側にトラフィックを返すことができます。



オプション２と４は不正解です。NATインスタンスはEC2インスタンス上でNATソフトウェアを利用する仕組みであり、NATゲートウェイの代わりに構成することが可能です。したがって、この構成も機能しますが、現在ではNATゲートウェイの利用が推奨されており、NATインスタンスを使うことは推奨されていません。

オプション３と４は不正解です。１つのAZのパブリックサブネットに１つNATを構成する設定では、データセンター障害やAZの停止に弱いため、冗長性を達成できません。
