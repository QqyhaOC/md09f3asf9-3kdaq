Auto Scalingによるインスタンス起動時に問題が発生すると、グループ内で実施されている1つ以上のプロセスを中断します。ユーザーはそのプロセスを任意に再開することができます。プロセスを中断することによって、ユーザーが設定上の問題を解析することができます。したがって、オプション２が正解となります。

ソリューションアーキテクトは、どのようにAWS Storage Gateway構成を設定すればよいでしょうか？

説明

このシナリオでは、AWS Storage Gatewayを利用してオンプレミス環境にあるストレージとAmazon S3とを連携して、拡張する構成方法が問われています。AWS Storage GatewayはAmazon S3バケットとオンプレミスストレージを接続して、バックアップを取得したり、ストレージ容量を拡張したりすることができるハイブリッドクラウド機能を提供しています。




AWS Storage Gatewayでは主に以下の３つのゲートウェイタイプを選択することができます。それぞれのゲートウェイタイプに応じて用途が異なるため、要件に応じて最適なタイプを選択できる必要があります。



さらにボリュームゲートウェイは補完型とキャッシュ型の２つに分かれるため、それぞれの違いを抑える必要があります。



今回は、頻繁にアクセスされるデータへの最適な応答時間を維持しつつ、ローカルストレージ容量の制約を回避するために、このローカルストレージをAmazon S3ストレージに拡張します。そして、その際はAmazon S3の標準ストレージタイプをプライマリーストレージとして利用していくことが要件となっています。



オプション１が正解となります。ストレージゲートウェイのキャッシュ型ボリュームを使用することで、頻繁にアクセスされるデータをキャッシュとしてローカル環境に保持しながら、S3をプライマリデータストレージとして使用できます。その際は、Amazon S3への移転スケジュールが設定されたスナップショットを利用してキャッシュ型ボリュームを構成して、ローカルストレージ容量の制約を回避するために、このレポジトリーをS3ストレージに拡張します。そして、キャッシュを取得する仕組みをオンプレミスストレージ側に設定することで、頻繁にアクセスされるデータへはキャッシュを通じて最適な応答時間を維持することができます。

キャッシュ型ボリュームは、オンプレミスのストレージインフラストラクチャをスケールする必要性を最小限に抑えます。同時に、アプリケーションは引き続き頻繁にアクセスするデータへの低レイテンシーなアクセスが可能となります。利用可能なストレージボリュームの最大サイズは 32TiBです。それを iSCSI デバイスとしてオンプレミスのアプリケーションサーバーにアタッチすることが可能です。このゲートウェイは保存データを Amazon S3 に作成されたストレージボリュームに保存し、最近読み込まれたデータはオンプレミスのストレージゲートウェイのキャッシュに保持して、バッファストレージにアップロードします。


オプション２は不正解です。ストレージゲートウェイの保管型ボリュームを使用すると、ローカルストレージをプライマリーとして利用して、そのデータを非同期に S3にバックアップします。これはプライマリーストレージにS3バケットを適用するという要件に合致していません。

オプション４は不正解です。仮想テープライブラリーはテープ形式のバックアップに利用されるものであり、今回の要件には関係がありません。

説明

オプション２が正解となります。デフォルトでは、EC2インスタンスのパブリックIPアドレスはインスタンスの停止後に解放されます。これによって、ドメイン名にマッピングされていた以前のパブリックIPアドレスが無効になりアクセスができなくなります。こうしたケースを避けるためには、EC2インスタンスにElastic IPを設定することが必要です。これによって、EC2インスタンスの再起動後も同じElastic IPアドレスが持続されることになり、Elastic IPアドレスに関連付けられたドメイン名を継続して利用することが可能となります。




オプション１は不正解です。Route53のルーティング先としてEC2インスタンスを指定している場合、EC2インスタンスのIPアドレスが変更されるとドメインの関連付けがエラーとなってしまいます。したがって、EC2インスタンスの再起動後にIPアドレスが変更された場合は、Route53のトラフィック設定が再度必要となります。しかしながら、今回のケースでは、再起動後のIPアドレスの変更が発生しないように対応することが本質的な解決策となるため、変更が発生した場合の事後対応として、Route53でのIPアドレスの再設定は解決策となっていません。

オプション３は不正解です。ELBのターゲットグループにEC2インスタンスを構成する場合は、IPアドレスではなく、EC2インスタンスを指定して構成します。したがって、EC2インスタンス再起動後にIPアドレスが変更されても、ELBの構成が失われることにはなりません。

オプション４が正解となります。Amazon EC2インスタンスはVPC内のサービスですが、Amazon S3はリージョンに設置するためVPC外のサービスとなります。したがって、デフォルトのネットワーク構成において、EC2インスタンスとS3バケット間で通信すると、インターネットを介して通信を行うことになり、データ転送コストが発生してしまいます。

VPCエンドポイントを使用するとインターネットを介さずに、VPC内のサービスからVPC外のAWSサービスに接続することができます。これにより、インターネットを通さずにEC2インスタンスからS3バケットへの通信を実施することができます。同一リージョン内の通信料金は無料であるため、通信料金は発生しなくなります。

VPCからAmazon S3 へのアクセスには、ゲートウェイエンドポイントとインターフェイスエンドポイント (AWS PrivateLink を使用) の 2 つのタイプの VPC エンドポイントを使用できます。

ゲートウェイエンドポイントはAWS ネットワーク経由で VPC から Amazon S3 にアクセスするためにVPCのルートテーブルにおいてルート指定が必要なゲートウェイです。ゲートウェイエンドポイントは、VPC にインターネットゲートウェイや NAT デバイスを通さずに、Amazon S3 および DynamoDB への信頼性の高い接続を提供します。 S3 や DynamoDB へ接続したいリソースが配置されている VPC に VPC エンドポイントを割り当て、ルートテーブルに ターゲットが VPC エンドポイントのルーティング を設定します。

インターフェースエンドポイントはプライベート IP アドレスを使用してVPC から Amazon S3 にリクエストをルーティングすることができるインターフェースタイプのエンドポイントです。ゲートウェイではないため、プライベート IP アドレスを利用した制御を行います。



オプション２も正解となります。以前はAmazon S3はインターフェースエンドポイントに対応していませんでしたが、2021年からAWS PrivateLink for Amazon S3という新機能が追加されました。AWS PrivateLink for Amazon S3 ではインターフェイスエンドポイントを利用して、VPCからAmazon S3バケットにアクセスができるようになります。



オプション１は不正解です。インターフェイス VPC エンドポイント (インターフェイスエンドポイント) を利用する場合は、 Amazon S3 へのアクセスをコントロールするエンドポイントポリシーをアタッチすることが必要です。

オプション３は不正解です。インターネットゲートウェイ経由でルーティングしてS3バケットにアクセスすることは可能ですが、画像をダウンロードするインターネットへのアウトバウンド通信の際にデータ転送コストが発生してしまいます。

オプション５は不正解です。ゲートウェイエンドポイントはパブリックIPアドレスを利用して、 Amazon S3 にアクセスします。
